# Создание виртуальных машин

Создаю 6 ВМ:
- 3 для PostgreSQL + Patroni;
- 3 для Etcd + haproxy.

По хорошему, развернуть haproxy на отдельной машине, но в Yandex Cloud не разрешается по стандартной квоте создавать более 6 машин.

# Установка PostgreSQL и Patroni
На ВМ `postgresql-1`, `postgresql-2`, `postgresql-3`:
Обновляю пакеты:
```bash
sudo apt update && sudo apt upgrade -y
```

Устанавливаю СУБД PostgreSQL 16:
```bash
sudo apt install postgresql-16 -y
```

Устанавливаю Patroni:
```bash
sudo apt install patroni -y
```

# Настройка haproxy
Настраиваю конфигурационный файл haproxy:
```bash
sudo nano /etc/haproxy/haproxy.cfg
```
```
global
        log /dev/log    local0
        log /dev/log    local1 notice
        chroot /var/lib/haproxy
        stats socket /run/haproxy/admin.sock mode 660 level admin
        stats timeout 30s
        user haproxy
        group haproxy
        daemon

defaults
        log     global
        mode    tcp
        option  tcplog
        option  dontlognull
        timeout connect 5000
        timeout client  50000
        timeout server  50000

frontend postgres_read_write
        bind *:5433
        default_backend master_nodes

frontend postgres_read
        bind *:5434
        default_backend replica_nodes

backend master_nodes
        mode tcp
        option httpchk GET /master
        http-check expect status 200
        default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions
        server postgres-1 10.130.0.20:5432 check port 8008
        server postgres-2 10.130.0.31:5432 check port 8008
        server postgres-3 10.130.0.29:5432 check port 8008

backend replica_nodes
        mode tcp
        option httpchk GET /replica?lag=100MB
        http-check expect status 200
        default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions
        server postgres-1 10.130.0.20:5432 check port 8008
        server postgres-2 10.130.0.31:5432 check port 8008
        server postgres-3 10.130.0.29:5432 check port 8008
```

 `global`
- **`log /dev/log local0`** – отправлять все логи (уровня info и выше) в системный syslog через `/dev/log` с меткой `local0`.
- **`log /dev/log local1 notice`** – дополнительно писать предупреждения и ошибки (уровень `notice`) с меткой `local1`.
- **`chroot /var/lib/haproxy`** – после запуска HAProxy ограничивает себя корневой файловой системой внутри `/var/lib/haproxy` (повышение безопасности).
- **`stats socket /run/haproxy/admin.sock mode 660 level admin`** – создаёт Unix-сокет для административного управления HAProxy (изменение весов, включение/выключение серверов). Доступен для пользователя и группы с правами чтения/записи.
- **`stats timeout 30s`** – максимальное время ожидания команд, отправленных в административный сокет.
- **`user haproxy`** – после старта процесс переключается на работу от пользователя `haproxy`.
- **`group haproxy`** – аналогично, группа процесса становится `haproxy`.
- **`daemon`** – HAProxy работает как фоновый демон.

`defaults`
- **`log global`** – использовать настройки логирования из секции `global`.
- **`mode tcp`** – работа на транспортном уровне (TCP-прокси), содержимое трафика не анализируется. Это необходимо для PostgreSQL, который использует бинарный протокол, а не HTTP.
- **`option tcplog`** – формат логов, оптимизированный для TCP-соединений (фиксируются длительность, объём данных и т.п.).
- **`option dontlognull`** – не записывать в лог соединения, в которых не было передано никаких данных (клиент подключился и сразу отключился).
- **`timeout connect 5000`** – таймаут установки TCP-соединения с сервером PostgreSQL (5 секунд).
- **`timeout client 50000`** – максимальное время неактивности со стороны клиента, после которого HAProxy разрывает соединение (50 секунд).
- **`timeout server 50000`** – максимальное время неактивности со стороны сервера PostgreSQL, после которого соединение закрывается (50 секунд).

 `frontend postgres_read_write`
- **`bind *:5433`** – HAProxy принимает подключения клиентов на порт 5433 на всех доступных сетевых интерфейсах.
- **`default_backend master_nodes`** – весь трафик, пришедший на этот frontend, направляется в бэкенд с именем `master_nodes` (где находится мастер PostgreSQL).

`frontend postgres_read`
- **`bind *:5434`** – слушает порт 5434 для входящих клиентских подключений.
- **`default_backend replica_nodes`** – трафик с этого порта уходит в бэкенд реплик (`replica_nodes`), распределяясь между ними.

 `backend master_nodes`
- **`mode tcp`** – режим TCP, как и в defaults.
- **`option httpchk GET /master`** – активирует HTTP-проверки состояния серверов. HAProxy отправляет GET-запрос по пути `/master` на порт, указанный в `check port` (8008). Этот endpoint предоставляет Patroni REST API.
- **`http-check expect status 200`** – сервер считается здоровым (UP), только если HTTP-ответ имеет код 200. Patroni возвращает `200 OK` исключительно для текущего мастера.
- **`default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions`** – параметры по умолчанию для всех серверов в этом бэкенде:
  - `inter 3s` – интервал между проверками (3 секунды).
  - `fall 3` – после трёх подряд неудачных проверок сервер помечается как DOWN.
  - `rise 2` – после двух успешных проверок сервер возвращается в состояние UP.
  - `on-marked-down shutdown-sessions` – при переходе сервера в DOWN все активные клиентские сессии на этом сервере немедленно обрываются.
- **`server postgres-1 10.130.0.20:5432 check port 8008`** – описание первого узла:
  - `postgres-1` – алиас сервера.
  - `10.130.0.20:5432` – IP и порт PostgreSQL, куда HAProxy будет направлять TCP-трафик.
  - `check` – включить проверки здоровья.
  - `port 8008` – проверки выполняются на порт 8008 (REST API Patroni) вместо основного порта 5432.
- Второй и третий серверы (`postgres-2`, `postgres-3`) описаны аналогично, с соответствующими IP-адресами.

### `backend replica_nodes`
- **`mode tcp`** – аналогично.
- **`option httpchk GET /replica?lag=100MB`** – HTTP-проверка идёт по пути `/replica` с параметром `lag=100MB`. Patroni возвращает `200`, если узел является репликой и её задержка относительно мастера не превышает 100 мегабайт.
- **`http-check expect status 200`** – только ответ с кодом 200 делает сервер доступным для приёма трафика.
- **`default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions`** – те же параметры проверок, что и для мастер-бэкенда.
- **Строки `server ...`** – те же три узла Patroni, но теперь они проверяются на пригодность как реплики. Балансировка между прошедшими проверку серверами осуществляется методом roundrobin (по умолчанию).
